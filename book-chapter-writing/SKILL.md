# Skill: 书稿章节写作（访谈→第一人称口述体）

> 触发场景：基于访谈录音/速记，为《AI落地之书》或类似项目写一篇5000字左右的书稿章节
> 最后更新：2026-06-09

---

## 触发场景

- 用户提供访谈原文（录音转文字/速记），要求产出书稿章节
- 体裁：第一人称（受访者视角）+ 书面语
- 典型产出：4000-6000字正文 + 扉页信息

---

## 每篇稿件的固定结构

每篇章节稿件由"扉页"和"正文"两部分组成：

### 扉页（老师介绍页）

按以下顺序排列：

1. **二级标题**：`## 老师介绍页`
2. **标题**：一级标题，文章大标题，简短有力。要求"小切口"——从一个具体意象/案例入手，不用概括性或抽象性的表述
3. **阅读理由**：1-3句话，不超过80字。告诉读者"为什么要读这篇"——落在核心洞察/论断上，最好是嘉宾自己的判断
4. **作者名**
5. **作者title**：一行以内，包含身份标签+从业年限+代表作（如有）
6. **商务照片占位**：`⚠️【待XX老师提供一张对外的商务照片】`

**阅读理由写作要求：**
- 不超过80字
- 落在嘉宾的核心洞察和论断上（最好是嘉宾自己的判断/金句）
- 不出现具体时间（如"2026年5月的访谈"），因为放在书里会显得过时
- 不罗列分散的效果数据
- 语言自然流畅，不生硬

### 正文

- 第一人称（受访者"我"视角）+ 书面语
- 开头直接进入叙事，不重复扉页已交代的信息
- 结构详见下方"阶段4写作规则"

---

## 前置条件

1. 【必选】访谈速记/录音转文字（非智能纪要）
2. 【必选】用户标注：哪几句话价值最大、一句话主线+核心读者、哪个案例值得放大
3. 【可选】受访者补充材料、公开文章

缺任何一项必选，不进入写作阶段。

### 主动素材比对（写稿前必做）

在动笔前，主动搜索嘉宾的公开发表内容（公众号文章、演讲、播客等），作为对照参考：
- 确认稿件是否遗漏了嘉宾自己反复强调的核心观点
- 确认稿件的论述角度是否与嘉宾公开表达一致（不能写出嘉宾自己都不认同的结论）
- 发现遗漏时主动向用户报告："嘉宾在XX文章里还强调了XX观点，稿件里没覆盖，是否需要补入？"
- 注意：公开文章是参考，不是写作样本。风格要做区分，不能变成改写/洗稿。

---

## 阶段4写作规则（10条）

### 一、体裁与风格

1. **体裁确认**：第一人称（"我"视角）+ 书面语 + 完整句式。
   - 口述体是叙事视角，不是语言风格。
   - 判断标准：念出来像播客/音频→不对。像人物特稿中的第一人称章节→对。
   - 不用第二人称碎句招呼读者（"你想想""你感受一下"）。
   - 不用"说起来有点绕""我尽量说清楚"等口播过渡。

### 二、结构与主线

2. **核心概念关系先定**：在框架阶段确认本篇核心概念之间的关系（并列/递进/包含/因果），与用户对齐后再动笔。
3. **主线优先用递进结构**：时间轴、因果链、由浅入深。避免平行分类（容易出逻辑漏洞）。
4. **案例数量≤4个**：超过4个认知任务分散，读者记不清每个案例证明什么。如需5个，必须有明确的递进层级。
5. **总结只出现一次**：放在所有案例讲完之后。中间用短过渡承接，不用长段总结（否则读者以为文章要结束了）。
6. **小标题规则**：
   - 全文4个左右，不超过5个
   - 先按字数均分（每段1500-2000字），再按语义微调切分点
   - 每个小标题3-8个字，关键词级别，要有信息增量
   - 不能只是复述段落内容（如"法务AI全景"太平），要让读者看到标题就捕捉到核心信息
   - 禁止句子式长标题
7. **同一论点不重复论证**：如果一个观点已经展开论证过（举了例子、给了数据），后文再涉及时一句话带过即可，不要换个框架再论证一轮。检查方法：全文搜索是否有两段在回答同一个问题。

### 三、案例写作

6. **每个案例写前一句话定位**：核心场景是什么？与用户对齐。（如"多点=生鲜出清"而非"产品选择"）
7. **零编造引语**：凡是打引号的对话/直接引语，必须能在原文中找到对应原话。找不到就不用引号，改为间接转述。
8. **行业术语和品牌品类要精确**：写前列一张"品牌→品类→行业"对照表，防止混用（如糕点≠面包，出清≠选品）。

### 四、过渡与节奏

9. **过渡靠内容承接**：
   - 不做目录预告（"以下几个案例说明"）
   - 不用编号标题（"案例一""案例二"）
   - 好的过渡：从上一个案例的结论自然延伸到下一个案例的场景
   - 加粗品牌名做视觉锚点即可

### 五、字数与补量

10. **字数标准：纯汉字+字母+数字**，不含标点和空格。每次报字数前先跑脚本确认。
    - 字数不够时优先加案例细节（场景、数字、结果），不加空泛论述
    - 补字数前先判断与主线的相关性，弱相关的不如把已有案例写厚

## 流程纪律

| 规则 | 说明 |
|------|------|
| 写完即核对原文 | 逐段标注来源（原文/公开资料/作者推演），发挥内容标注⚠️让用户确认 |
| 每次大改后输出完整版 | 不只贴片段，防止版本混乱 |
| 参考资料≠写作样本 | 用户给的任何材料，先确认在本篇中的角色 |
| 受访者已有公开文章 | 必须与本篇做风格区分，不能混为参考或模仿对象 |
| 脱敏规则 | 受访者明确不能提名的品牌/人物，全文标注并脱敏处理 |
| 多位受访者交叉引用 | 每段明确归属哪位受访者的原话，不混淆来源 |
| 标题承诺即时兑现 | 标题用了某个具体意象（如茶叶蛋），正文第二段之内必须出现这个画面，不让读者等 |
| 结尾不是附加段 | 正式结尾要有力量感和收束感，行动指南/落地建议放在结尾之前作为独立小节 |

## 阶段5自查清单（给用户看之前必须完成）

### A. 逻辑链自检（最高优先级）

写完初稿后，**逐对检查相邻段落**：
- 上一段结尾和下一段开头之间，是否有因果或递进关系？
- 如果没有，必须补过渡句或调整段落顺序
- 对于不确定的衔接处，标注`⚠️[过渡待确认]`
- 自检后主动向用户报告："共发现X处衔接需要补充，已补/已标注"

**逻辑断层的常见模式**（从实战中提炼）：
- "问题→解法"之间缺因果（为什么这个问题能被这个解法解决？）
- "论述→例子"之间缺引入（为什么突然开始聊这个例子？）
- "例子→下一话题"之间缺收束（上个例子说明了什么，跟下段有什么关系？）
- 同一概念重复论证两轮（换了框架但素材和结论高度重叠）

### B. 原文忠实度检查

1. 是否有偏离主线的表述
2. 是否有不忠于访谈原文的内容（逐案例核对）
3. 是否有不忠于事实的表述
4. **每个关键论断标注来源**：`[原文]` 或 `[推理]`，推理处额外标注推理依据
5. 推理内容不得以断言语气写出，需用"在我看来""这意味着"等限定

### C. 节奏与可读性

6. 每个抽象论点有没有可感知的例子或画面
7. **节奏规则**：每个核心论点后面必须跟至少一个具体例子或画面，不能纯论述连续超过3段
8. 有没有行业黑话没解释、有没有绝对化表述

### D. 格式规范

9. **标点符号统一**：引号必须是中文弯引号「"  "」，英文标点不得出现在中文语境中
10. **ASR专有名词扫描**：源头为语音转文字时，逐一确认所有品牌名、人名、地名的拼写是否正确
11. **语言洁癖**：禁用→箭头符号、禁用「首先/其次/最后」三段式、禁用无来源的断言、不用AI味的列举句式

## 常见错误和排查

| 错误 | 原因 | 解法 |
|------|------|------|
| 写成第三人称评论稿 | 没确认体裁 | 起笔前确认：谁在说话 |
| 写成音频口播稿 | 把"口述体"理解为"口语化" | 口述体=叙事视角，不是语言风格 |
| 案例之间逻辑矛盾 | 核心概念关系没定清楚 | 回到框架阶段重新对齐 |
| 字数虚高 | 含标点统计 | 跑脚本：纯汉字+字母+数字 |
| 编造引语 | 为了生动加了对话 | 回原文找，找不到就删 |
| 读者中途以为文章结束 | 总结段出现太早 | 总结只在最后出现一次 |
| 归纳出原文不存在的递进路径 | 写作时为了逻辑清晰自己编了顺序 | 逐段回速记核对，原文没说的不能当成受访者的结论 |
| 相邻段落自相矛盾 | 指的是不同对象但表述上没分清 | 写完后逐对相邻段落检查，对比关系必须在文字层面写明 |
| 无来源断言 | 受访者口述中提到但没有出处 | 改模糊表述或标⚠️待核实，不用断言语气 |
| 标题承诺落空 | 标题用了具体意象但正文要等很久才出现 | 第二段之内兑现标题承诺 |
| 用箭头/列举等AI味句式 | 工具习惯带入书面语 | 书稿中禁用→符号，递进用自然语言表达 |
| 报告体「第一…第二…第三…」 | 把多观点等同于列表 | 用叙事过渡衔接，不用编号 |
| 冒犯性表达（居高临下描述某群体） | 写作者视角代入 | 从被描述群体自身视角出发叙述 |
| 技术细节堆砌（算力/架构/芯片） | 以为读者需要了解实现 | 只呈现用户侧可感知的结果 |
| 术语不解释 | 以为读者有行业背景 | 每个术语首次出现必须行内通俗解释 |
| 编造具体数字/地名 | 为了生动补充细节 | 原文没有的具体信息不加，标⚠️向作者确认 |
| Overwrite 飞书覆盖用户编辑 | 没先 fetch 确认飞书端最新版 | 修改前必须 fetch，用户确认后才覆盖 |
| 改了用户没要求改的地方 | 顺手优化 | 只改用户明确指出的那个点，其他问题另提 |
| 省略逻辑背景导致读不懂 | 以为读者能自行补充前提 | 关键论断前必须铺因果链（为什么→所以） |

## 验证方式

- 字数达标（纯汉字+字母数字，目标±10%）
- 阶段5自查清单5项全过
- 用户审稿无重大结构性修改意见
- 所有待确认信息已标注留空

## 回撤方案

- 每次大改前保留上一版完整文本
- 如果新版方向走偏，可以回退到上一个用户确认过的版本继续迭代

## 初稿交付流程（给用户看之前）

### Step 1: 骨架确认

动笔前先输出一份「段落骨架」——每个段落用一句话写清：
- 这段的功能是什么（开场画面/铺数据/解释技术/过渡/收束）
- 读者在这段应该获得什么
- 这段的内容来源（原文哪段/哪个时间戳）

骨架经用户确认后再写全文。

### Step 2: 原文交叉验证

全文写完后，逐段回原始速记核对：
- 每段归纳性结论 → 标注原文出处（时间戳或引文）
- 如果某段是写作加工（原文没有明确说过这个顺序/递进/结论）→ 标⚠️「写作加工」
- 如果某段引用了具体数字/机构/事件但速记里没有明确出处 → 标⚠️「待核实」

### Step 3: 输出自查报告

初稿交付时附带一份简短自查报告：

```
## 自查报告
- 原文直接支撑的段落：X段
- 写作加工的段落：X段（已标注⚠️）
- 待核实信息：X项（已标注⚠️）
- 逻辑自洽检查：通过/有X处需注意
- 标点/ASR扫描：通过/有X处已修正
```

用户只需审核「写作加工」和「待核实」部分，不需要逐字通读。

## 文档输出规范

初稿完成后，**直接输出飞书文档链接**，不给用户 md 文件。

标准交付动作：

1. **写入飞书文档**：用 `lark-cli docs +create --title "【初稿】XX - AI落地之书" --markdown "$(cat file.md)" --as user --api-version v1` 创建
2. **交付物是链接**：把飞书文档 URL 发给用户，不发本地 md 文件路径
3. **后续修改同步更新飞书**：每次精调后用 `lark-cli docs +update --mode overwrite` 同步到飞书文档
4. **标点检查**：写入前跑一遍全文标点扫描（引号、括号等），确保无英文标点残留
5. **附带确认清单**：文档末尾附上待作者确认的信息清单
6. **附带插图素材建议**：3-5张插图建议（含插入位置、素材内容描述、来源建议），素材不求多但求能还原一线真实场景

## 飞书文档协作纪律

### 核心原则

**飞书文档是唯一权威版本**。用户会随时在飞书上做修改，本地文件不是权威源。

### 读取飞书文档

用 `lark-cli docs +fetch --doc "<url>" --as user --api-version v1` 读取飞书文档当前内容。**修改前必须先 fetch**，确认飞书端当前状态。

### 写入/同步纪律

| 规则 | 说明 |
|------|------|
| 初稿创建用 `+create` | 第一次交付时创建新文档 |
| 后续修改用 `replace_range` 定点替换 | **不要 overwrite 覆盖全文**，会丢失用户随手修改 |
| 定点替换语法 | `--mode replace_range --selection-with-ellipsis "开头文本...结尾文本" --markdown "新内容"` |
| 插入新段落 | `--mode insert_before --selection-with-ellipsis "定位文本"` |
| 用户在飞书上的编辑是权威版本 | 用户改了飞书，以飞书为准 |
| 仅在用户明确要求时才 overwrite | 如标点全文替换等批量操作，用户明确同意后才可覆盖 |
| 用户说"不要动"就停手 | 不要趁机改其他地方 |

### 修改范围纪律

- 用户给一个反馈点，**只改那一个点**，不顺手动其他段落
- 如果发现其他问题，单独提出让用户决定，不自行修复
- 修改后报告：改了哪里、改了什么、为什么改

## 写作风格红线（从实战中提炼）

### 禁止报告体

- 不用「第一…第二…第三…」「一是…二是…三是…」编号结构
- 不用「以下从三个维度展开」等预告式导语
- 多个观点之间用叙事过渡衔接，不用分类标签

### 禁止冒犯性表达

- 不用「让XX群体接受/理解/学习…」——暗含教育者视角
- 不用「教育成本」「认知门槛」等居高临下的词
- 正确做法：从被描述群体自身视角出发叙述

### 禁止技术堆砌

- 读者不需要知道实现细节（算力多大、用什么芯片、什么架构）
- 只呈现用户侧可感知的结果和体验
- 技术名词必须有行内解释（括号或破折号），无法解释的就删掉

### 行业术语处理

- 每个行业术语第一次出现时，用括号/破折号给出通俗解释
- 如果一个术语解释起来超过10个字，考虑直接用通俗说法替代
- 列一张「术语→通俗表达」对照表，写前确认

### 结尾和标题

- **结尾**：用嘉宾自己的核心判断收束，要有力量感，不要泛泛总结
- **标题/阅读理由**：落在嘉宾的核心判断上，不用概括性描述
- 判断标准：如果把作者名遮掉，这句话还能不能成立？——能，就是好判断

### 标题/阅读理由去AI味规则

写完后必须用以下检查项逐条过：

1. **有没有"不是…而是…"句式？** → 禁用。这是最典型的AI对仗句式。
2. **有没有"真正的…是…"句式？** → 禁用。同上。
3. **有没有具体的人？** → 必须有。不能是纯抽象判断。
4. **有没有具体的事/成绩/数字？** → 最好有。
5. **有没有具体的洞察（而非概括）？** → 洞察要是嘉宾本人说的，不是你归纳的。
6. **读起来像不像一个人说的话？** → 要口语化，像朋友间安利："这个人做了一件事，他发现了一个道理"。
7. **时效性检查**：放到明年看会不会过时？如果会，用"永恒命题"替代"阶段描述"。

## 原文忠实度：区分信息来源

访谈类稿件可能有多个信息来源，必须严格区分：

| 来源类型 | 可信度 | 处理方式 |
|---------|--------|---------|
| 访谈速记/录音转文字 | 最高 | 可直接引用，标注时间戳 |
| 受访者提供的补充材料（PDF/PPT等） | 高 | 可用，但注明来源非口述 |
| 公开资料（官网/新闻） | 中 | 交叉验证后可用 |
| 写作加工（为逻辑通顺而补充的衔接） | 低 | 必须标⚠️「写作加工」 |

**关键纪律**：
- 不编造具体数字（"花了大几万"→原文没说具体金额就不写具体金额）
- 不编造地名/省份/具体场景（原文没提到就不加）
- 受访者刻意模糊的信息（如不愿透露客户名），尊重其意愿，不推测
- 事实核查时区分「访谈中说的」vs「PDF材料中写的」——两者口径可能不一致
